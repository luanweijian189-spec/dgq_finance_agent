## 一、会议纪要延伸讨论（现状梳理）

### 用户问题

1. 输入的信息可能很多，需要一个模型识别股票名和内部的逻辑。这功能目前是缺失的。
2. 内部的数据流逻辑需要说清楚，上次没说清楚。目前这个系统的主要功能是：
   1. 数据输入：股票消息批量导入；资讯录入；针对推荐人进行手动打分等。这是一条功能链。
   2. 股票每天的信息更新，这个是一条数据链，从获取数据，到内部判断，到结论输出，到推荐人打分等，这条完整的数据链路说清楚。
3. 针对内部的数据流逻辑，距离最终稳定实盘和性能提升，还需要做很多事情：
   1. 比如输入源的正规化？
   2. 比如逻辑推理的数据参考完善。
   3. 比如内部判定逻辑。
   4. 比如荐股人打分机制等。
   我希望你把现状说清楚，然后针对目前可以提升的地方罗列清楚，给出你的解决方案（不要写代码，我们先讨论方案）。

### 回答整理

#### 1. 总体判断

当前系统已经具备一个可运行的 MVP 骨架，覆盖了：

- 输入采集
- 股票池管理
- 每日评估
- 荐股人打分
- 日报与告警

但它还不是稳定实盘系统，核心短板在于：

- 输入结构化识别不足
- 逻辑证据层不完整
- 判定链路解释性不足
- 权限、部署、登录、多用户机制未正式完成
- 资金流追踪 Agent 尚未建立

#### 2. 输入识别现状

当前输入识别并非完全没有，而是“规则解析优先”的初级版：

- 用正则提取股票代码
- 用关键词判断是否为荐股意图
- 从句子中抓取一段逻辑文本

现有能力对标准格式消息有效，但对以下情况处理较弱：

- 只说股票名不说代码
- 多股票混杂
- 聊天上下文引用
- 图片/OCR 内容
- 半结构化研究笔记
- 复杂逻辑拆解（催化、风险、时效性、时间窗口）

#### 3. 托管与登录现状

当前系统已经具备可部署后端：

- FastAPI
- SQLAlchemy + Alembic
- Web 管理后台
- PostgreSQL 适配

但当前没有正式的登录/权限体系，仍偏单机内部工具。

建议部署形态：

- 云服务器
- Docker Compose
- PostgreSQL
- Nginx + HTTPS
- 登录认证（JWT/Session）
- 角色划分：管理员 / 研究员 / 只读

#### 4. DGQ 核心需求现状

目标是围绕“外来信息逻辑 + 当前价格位置 + 人的历史质量”做股票排序。

当前已经具备初版骨架：

- 股票池
- 每日更新
- 股票日评分 SQS
- 荐股人评分 RRS
- AI 智能分析与次日预测
- 本地 RAG / 记忆库

但仍存在：

- 输入质量不稳定
- 证据参考不足
- 价格位置理解不够深
- 资金面未纳入核心排序
- 模型不可用时会回退到 fallback 逻辑

#### 5. OpenClaw / 通知机制

当前已具备：

- OpenClaw 命令入口
- WeChat webhook 入口
- 告警订阅链路

但仍需要补强：

- 用户身份绑定
- 任务执行状态
- 消息回执
- 失败重试
- 消息分级模板

#### 6. QQ 接入现状

QQ 目前没有成型实现。现阶段更建议优先提升输入标准化、评估质量与生产化能力，而不是优先加新入口。

#### 7. 模型 API 方向

当前模型层是 OpenAI 兼容接口设计，因此理论上可以切到：

- Minimax 2.5
- DeepSeek
- 智谱
- Kimi
- 本地 vLLM / Ollama

建议采用双层模型：

- 轻模型：输入识别、抽取、标签化
- 强模型：逻辑推理、风险判断、交易结论

#### 8. 输出逻辑细节

当前已经有日报、状态页、预测结果，但解释性还不够。

后续应补齐：

- 为什么这只股票排前面
- 是价格、逻辑还是新闻推动
- 关键证据有哪些
- 风险点有哪些
- 什么条件会推翻当前结论

#### 9. 资金追踪 Agent

当前系统尚未真正实现资金追踪层。后续若要接近实盘，应补充：

- 主力资金
- 北向资金
- 龙虎榜
- 大单净流入
- 板块轮动

未来排序模型建议升级为：

$$
排序分 = 逻辑质量 + 价格位置 + 资金确认
$$

#### 10. 分阶段解决方案

建议分四阶段推进：

1. 输入识别重构
   - 统一输入 schema
   - 规则 + LLM 双通道解析
   - 人工确认队列
   - 股票名映射台

2. 每日更新链稳定化
   - 数据状态可追踪
   - 异常原因可解释
   - 输出证据可回溯

3. 排序逻辑升级
   - 加入价格位置
   - 加入资金面
   - 加入推荐人风格权重
   - 输出状态机而非单纯分数

4. 生产化部署
   - 云端部署
   - 登录认证
   - 多用户角色
   - 手机访问
   - 稳定通知系统

## 二、关于“每天更新逻辑”的追问

### 用户问题

描述下，目前针对股票池里股票进行每天更新的详细逻辑。

### 回答整理

#### 1. 调度入口

- 是否启用定时任务由配置决定。
- 默认工作日 15:30 执行一次每日评估。
- 启动时注册 APScheduler 定时任务。

#### 2. 每日更新主流程

每日任务主函数会按如下顺序执行：

1. 先复盘今天到期的前一日预测。
2. 遍历所有 `tracking` 状态的股票。
3. 通过行情数据源获取当日快照：收盘、高点、低点、涨跌幅、回撤、Sharpe、量化因子。
4. 通过新闻数据源验证原始推荐逻辑是否仍被近期信息支持。
5. 计算股票日评分 `SQS`。
6. 写入/更新当日 `DailyPerformance`。
7. 生成下一交易日的 AI 预测。
8. 刷新所有荐股人的可靠性分数 `RRS`。
9. 生成当日日报。
10. 推送简版战报。

#### 3. 当前系统为何会出现“无评分”

主要有两类原因：

- 股票仍处于 `pending_mapping`，没有真实证券代码。
- 当前配置使用的是 `mock` 数据源，或数据源未返回有效真实行情。

### 回答整理

#### 1. 关于定时任务启动

“定时任务启动”指的是：服务启动后，会根据配置自动注册后台任务，由调度器在指定时间执行。

当前默认配置：

- 每日评估：工作日 15:30 执行
- 新闻扫描：工作日 9:00 到 15:59 每 20 分钟执行一次

这两个默认值来自系统配置。

#### 2. 股票现状的数据源与字段含义

当前行情数据源支持两种：

- `mock`
- `baostock`

如果使用 `baostock`，系统会读取当日或最近交易日的日线行情，并计算内部快照字段：

- `close`：收盘价，即该交易日最后成交价
- `high`：最高价，即该交易日盘中最高成交价
- `low`：最低价，即该交易日盘中最低成交价
- `pnl_percent`：涨跌幅
- `max_drawdown`：日内高点到低点的回撤幅度
- `sharpe_ratio`：当前系统内部用涨跌幅与回撤粗算出的风险收益比
- `market_cap_score`：市值维度分，目前是简化因子
- `elasticity_score`：弹性分，来自日内高低波动幅度
- `liquidity_score`：流动性分，来自成交额规模

#### 3. 关于“运行一下程序更新股票”

已经支持手动触发一次更新。

手动更新的内容包括：

- 扫描一次新闻候选
- 跑一次股票池每日评估
- 刷新荐股人分数
- 生成日报

#### 4. 关于前端按钮

前端可以加，而且本次已经加上“一键抓取并更新”按钮。

该按钮会触发：

1. 先做一次新闻扫描
2. 再做一次股票池评估
3. 最后刷新页面并显示执行结果摘要

#### 5. 新闻/信息验证原逻辑的当前做法

当前新闻数据源支持四种：

- `mock`
- `tushare`
- `webhook`
- `sites`

其中：

- `mock`：简单按关键词判断逻辑是否成立
- `tushare`：拉取当日新闻标题，检查股票代码或逻辑关键词是否命中
- `webhook`：把校验请求发给外部服务，由外部返回 `validated`
- `sites`：从站点白名单抓首页及文章详情，计算语义与关键词匹配分

`sites` 的核心流程：

1. 抓取白名单财经网站首页 HTML
2. 提取站内文章链接
3. 用标题和逻辑文本做语义相似度打分
4. 再抓详情页正文做关键词与代码命中
5. 达到阈值则认为“逻辑被新闻支持”

同时它还支持“新闻发现新股票候选”的流程：

- 从文章中提取股票代码
- 识别事件类型（业绩/订单/政策/风险）
- 计算发现分
- 进入新闻候选池

#### 6. 股票评分机制（SQS）

当前股票评分由三部分组成：

$$
SQS = 0.6 \times 收益质量 + 0.2 \times 逻辑验证 + 0.2 \times 量化因子
$$

其中：

- 收益质量分：由涨跌幅、峰值收益、Sharpe、回撤共同计算
- 逻辑验证分：若新闻/信息支持逻辑，则为 100，否则为 45
- 量化因子分：由市值、弹性、流动性三项取平均

然后系统按最终分数给出简单结论：

- `>=80`：重点持有
- `<50`：谨慎观察
- 其余：继续跟踪

#### 7. 股票预测逻辑

当前预测逻辑是“结构化 LLM 决策优先，失败则 fallback”：

输入包括：

- 股票代码与名称
- 原始推荐逻辑
- 当日评分
- 当前收益
- 最大回撤
- 本地记忆/RAG 召回内容

模型输出结构化字段：

- `direction`：`up/down/sideways`
- `confidence`
- `horizon_days`
- `thesis`
- `invalidation_conditions`
- `risk_flags`
- `evidence`

如果模型不可用，则退化为：

- `direction=sideways`
- `confidence=0.35`
- `predicted_by=fallback`

#### 8. 荐股人打分机制（RRS）

荐股人分数是基于其历史推荐结果按时间衰减聚合得到。

核心参考维度：

- 历史收益率
- 历史回撤
- 命中率（系统里目前用收益超过一定阈值近似）
- 时间衰减（越近的推荐权重越高）

系统输出一个 `reliability_score`，并给出标签：

- 高胜率短线选手
- 稳定输出型
- 观察名单
- 高风险信号源

## 四、2026-03-07 补充问答：收益率来源与生产升级方案

### 用户问题

1. 东山精密和兴森当天收益率是哪里来的？为什么感觉数据不对？3 月 7 日拿到的数据是什么数据？3 月 7 日是周六，周五最终的收盘收益率应为 +2.96%。
2. 面向最终的生产目标，给我一个升级完善方案，我可以花钱购买云服务和 API，考虑我们的会议纪要和目前现状，帮我设计下。

### 回答整理

#### 1. 为什么 3 月 7 日的收益率看起来不对

当前你本地最近运行时使用的是：

- `MARKET_DATA_PROVIDER=mock`
- `NEWS_DATA_PROVIDER=mock`

所以当天看到的东山精密、兴森科技收益率，并不是真实行情，而是模拟数据源根据“股票代码 + 日期”生成的伪数据。

也就是说：

- 3 月 7 日是周六
- 系统并没有拿真实周五收盘
- 而是 `mock` provider 按日期 `day=7` 直接算出了一个模拟 `pnl_percent`

因此你看到：

- 东山精密 `+4.0%`
- 兴森科技 `+2.0%`

这两个值只是演示值，不是交易所真实结果。

#### 2. 当前 `mock` 收益率的生成方式

`mock` 行情源会根据股票代码字符和交易日日期做一个种子，然后计算：

- `close_price`
- `high_price`
- `low_price`
- `pnl_percent`
- `max_drawdown`

其中 `pnl_percent` 当前是：

$$
pnl\_percent = (seed \bmod 25) - 8
$$

所以它不是行情接口回来的真实涨跌幅，而是为了让系统调试时“每次都有数据”。

#### 3. 如果切换到真实行情会怎样

如果切换到 `baostock`：

- 系统会尝试取目标日期当天日线
- 如果当天没有数据（如周六、周日、节假日）
- 会回退查近 14 天数据
- 最终取最近一个有效交易日的日线作为结果

也就是说：

- 在 3 月 7 日（周六）运行
- 正常应该拿到 3 月 6 日（周五）最近一根有效日线

这是生产环境应采用的行为。

#### 4. 为什么你当前看到的结果与周五收盘不一致

根因不是评分公式，而是：

1. 当前运行环境是 `mock`
2. 手动按钮调用的也是当前服务进程对应的 `mock` provider
3. 所以 3 月 7 日显示的是模拟收益率，不是真实周五收盘涨跌幅

#### 5. 面向最终生产目标的升级方案

建议目标架构分为 6 层：

1. 输入接入层
2. 结构化识别层
3. 研究与证据层
4. 每日评估与排序层
5. 交互与通知层
6. 平台与运维层

##### 5.1 输入接入层

目标：统一接各种来源的消息与研究材料。

建议接入：

- 微信转发入口
- 手机 Web 录入
- 批量文本导入
- API 导入
- 研究员后台录入

所有输入统一进入同一个标准事件结构：

- 来源渠道
- 发送人
- 时间
- 原始文本
- 附件/图片/OCR 结果
- 解析状态
- 人工确认状态

##### 5.2 结构化识别层

目标：把原始消息转成可计算对象。

建议采用“规则 + LLM + 人审队列”三层：

- 规则层：股票代码、明显荐股语句、基础时间识别
- LLM 层：抽取股票名、逻辑、催化、风险、时间窗口、结论类型
- 人审层：低置信消息进入待确认队列

产出统一结构：

- 股票代码/股票名
- 推荐类型（推荐/跟踪/研究/噪音）
- 核心逻辑
- 催化事件
- 风险点
- 观察周期
- 置信度

##### 5.3 研究与证据层

目标：给后续评估和模型推理提供充分证据。

建议建设 5 类证据：

- 原始推荐记录
- 新闻事件流
- 研究笔记/RAG
- 行情与量价结构
- 资金流与板块信息

建议外部能力采购：

- 行情：Tushare Pro / 聚宽 / 东方财富接口 / Wind（预算允许时）
- 新闻：Tushare + 自建站点白名单抓取 + 交易所公告
- 模型：Minimax 2.5 或其他国内强模型 API

##### 5.4 每日评估与排序层

目标：把“逻辑 + 价格 + 人 + 资金”合并成稳定排序系统。

建议把现有单一 SQS 升级为四段式评分：

$$
总分 = 逻辑强度 + 价格位置 + 资金确认 + 信息源权重
$$

其中：

- 逻辑强度：原始逻辑是否被新证据强化或削弱
- 价格位置：是否处于合适的介入/观察位置
- 资金确认：成交额、主力流、板块联动是否支持
- 信息源权重：荐股人历史风格与命中能力

同时增加股票状态机：

- 新进入池
- 重点观察
- 可跟踪
- 风险升高
- 退出跟踪

##### 5.5 交互与通知层

目标：让系统能真正服务日常使用。

建议建设：

- Web 仪表盘
- 手机适配页面
- 一键刷新
- 单票详情页
- 预测复盘页
- OpenClaw / Telegram / 企业微信通知

通知分级：

- 日报
- 风险告警
- 逻辑失效告警
- 新催化提示
- 排名跃升提醒

##### 5.6 平台与运维层

目标：达到可长期在线运行的生产能力。

建议配置：

- 云服务器 1~2 台
- PostgreSQL
- Redis
- 对象存储（放附件/OCR/报告）
- Docker Compose 或 K8s（后期）
- Nginx + HTTPS

## 五、2026-03-07 新一轮需求整理：本地优先的下一阶段方案

### 用户要求

1. 输入识别引入 LLM。
2. 程序运行机制：
   1. 可以人工手动触发，比如目前前端页面。
   2. 可以定时触发，比如半小时触发一次。有新的结论后，把消息通过某种形式发送到用户手里（接口待定，后续可能通过 OpenClaw 驱动通讯软件）。
   3. 后续可以考虑新的触发条件。
3. 信息源问题：
   1. 可以每天盯盘实时数据，希望获取更专业的数据（日 K、量、价等全部技术分析的专业实盘数据），后续需要调研专业、高频的数据接口。
   2. 可以获取目标新闻网站，对行业有影响的、有价值的新闻，并进行信息提取。
4. 逻辑分析：
   1. 想让模型效果好，需要做好输入管理和模板设计。输入不应只有简短信息，还应包含历史数据、新闻数据、模型历史判断、方法论、原始逻辑和中间补充逻辑。资讯录入既可能是某只股票，也可能是通用宏观新闻。
   2. 每日逻辑分析不能只输出指标，要输出人能看懂的逻辑。
5. 迭代机制：
   1. 模型本身可能无法迭代，真正可迭代的核心是策略。

### 方案整理

#### 1. 输入识别：引入 LLM，但不能只靠 LLM

建议采用三层识别机制：

1. 规则快筛层
   - 股票代码识别
   - 人名/时间/消息结构识别
   - 明显推荐语句识别

2. LLM 结构化抽取层
   - 抽取股票名 / 股票代码
   - 抽取核心逻辑、催化、风险、时间窗口
   - 判断消息类型：荐股 / 跟踪 / 研究 / 宏观 / 噪音
   - 输出结构化 JSON

3. 人工确认层
   - 低置信度消息进入待确认队列
   - 股票名无法映射时进入待映射
   - 多股票混合时可人工拆分

目标：

- 不丢消息
- 不把噪音直接写成推荐
- 尽量把输入转成后续能计算的结构化对象

#### 2. 程序运行机制：改为“多触发器驱动”

建议统一成一个任务调度框架，触发源包含：

1. 手动触发
   - 前端按钮
   - 命令接口
   - API 手动调用

2. 定时触发
   - 半小时新闻扫描
   - 日终/盘后全量评估
   - 午间轻量复检

3. 条件触发（后续）
   - 某只股票价格异常波动
   - 新闻新增高价值事件
   - 逻辑失效条件命中
   - 推荐人信号与行情共振

输出机制建议统一成：

- 生成结构化结论
- 写入数据库
- 命中推送条件时，再交给通知层发送

当前先本地化，通知层可以先只保留：

- 页面提示
- 日志输出
- 本地命令查看

后续再接 OpenClaw / 通讯软件。

#### 3. 信息源：拆成“行情源”和“新闻源”两条线

##### 3.1 行情源

短期本地调试阶段：

- 继续使用 `baostock` 做日线级别验证

中期升级方向：

- 调研更专业的数据接口
- 获取更完整的：
  - 开高低收
  - 成交量
  - 成交额
  - 分时/分钟级数据
  - 板块联动
  - 技术指标所需原始序列

目标不是只拿一个收盘涨跌幅，而是形成“可用于技术分析和策略判断的完整价格序列”。

##### 3.2 新闻源

新闻建议分两类：

1. 个股/行业直接相关新闻
2. 宏观/政策/产业链通用新闻

处理流程建议：

- 抓取候选新闻
- 提取主题、行业、相关股票、事件类型、情绪方向
- 判断是“强相关个股事件”还是“通用宏观事件”
- 存入统一知识层

这样前端资讯录入就不只是“一个文本框”，而是：

- 既支持 stock-specific 资讯
- 也支持 macro/general 资讯

#### 4. 逻辑分析：核心不是多指标，而是“输入管理 + 模板设计 + 证据组织”

建议把模型输入拆成 5 类：

1. 原始逻辑
   - 当初为什么推荐

2. 后续补充逻辑
   - 中间新增的研究观点
   - 新的催化与反证

3. 价格/技术面数据
   - 日 K
   - 量价结构
   - 趋势与波动

4. 新闻/宏观/行业事件
   - 个股新闻
   - 行业新闻
   - 宏观新闻

5. 历史判断与方法论
   - 模型过去如何判断
   - 过去判断对错如何
   - 当前策略模板与偏好

然后让模型输出两层内容：

1. 结构化层
   - direction
   - confidence
   - evidence
   - invalidation

2. 人话层
   - 用研究员看得懂的话解释：
     - 为什么继续看好/转中性/转谨慎
     - 新增了什么证据
     - 哪些风险在变强
     - 接下来重点看什么

目标：日报不只是“分数表”，而是“人能读懂的逻辑分析”。

#### 5. 迭代机制：未来真正能迭代的是策略，不只是模型

这是非常关键的一点。

建议把系统拆成：

- 模型层：负责理解与表达
- 策略层：负责规则、排序、状态转移、决策框架

可持续迭代的重点应该是策略层，包括：

1. 输入模板怎么设计
2. 哪些证据进入上下文
3. 哪些新闻算强催化
4. 哪些价格位置值得重点跟踪
5. 推荐人权重如何参与排序
6. 哪些情况下结论应自动降级

建议建立策略迭代闭环：

1. 每天产出判断
2. 次日/阶段性复盘结果
3. 找出哪些判断命中、哪些失效
4. 调整策略参数/模板/状态机
5. 再进入下一轮迭代

也就是说：

$$
可迭代核心 = 策略 + 输入管理 + 证据组织 + 复盘机制
$$

而不只是“换一个更大的模型”。

#### 6. 本地优先的下一步执行顺序

建议按这个顺序推进：

1. 输入识别引入 LLM
   - 先做结构化抽取

   - 展示价格、新闻、逻辑、预测、复盘

- 长源东谷 → `300181`

这和真实代码不一致。

说明当前如果完全依赖模型直接给代码，风险仍然较高。

更稳的做法应是：

- 模型优先抽“股票名”
- 系统侧再用本地股票字典 / 查询接口做二次映射

##### （4）当前尚缺“名称候选 → 真实股票字典映射”的强校验层

虽然系统现在已经支持：

- 只有股票名时，若数据库已有同名股票，可映射真实代码

但它还不具备：

- 一个完整 A 股名称词典
- 名称模糊匹配
- 别名/简称映射
- 非 A 股实体过滤
- 多候选歧义消解

而这恰恰是输入阶段走向完备所必须补的一层。

### 当前结论：输入识别这部分能不能开始测？

结论是：

> **可以开始测，而且现在已经值得进入“系统性测试阶段”了。**

因为它已经不再是“完全不可用”的状态，而是：

- 单标的清晰荐股句式：已经基本能跑通
- 股票名映射：已经具备初版能力
- 研究 / 宏观 / 跟踪更新分类：已经能分出来

但是如果问：

> 输入阶段的功能现在是否已经“比较完备”？

那答案是：

> **还不能说比较完备，只能说“主框架已成立，已进入可系统测试但仍需补强”的阶段。**

### 最准确的成熟度判断

如果按成熟度分层：

1. 不可用
2. 可演示
3. 可系统测试
4. 可稳定内测
5. 可生产使用

那么当前“输入识别引入 LLM”这部分，更接近：

> **第 3 阶段：可系统测试**

还没完全到：

- 稳定内测
- 更不用说生产级输入引擎

### 当前最需要补的输入阶段能力

针对这次 `input_test.md` 的暴露结果，下一步最该补的是：

#### 1. A 股股票名称字典与自动映射层

目标：

#### 2. 非股票实体过滤

#### 2. 当前仍未完全补齐的输入短板

如果按真正稳定使用标准看，仍然缺 5 类关键能力：

##### （1）单票深度研报的主标的抽取

像：

- 科新机电深度
- 智立方扩产链
- 卓胜微硅光

这类文本虽然在说某只股票，但不一定带强推荐动作词，当前仍可能进不了推荐主链路。

##### （2）标题型 / 纪要型文本的主语继承

很多研报是：

- 标题先给股票名
- 后文多段都默认围绕这个股票展开

当前系统对这种“标题确定主语，后文省略股票名”的理解还不够强。

##### （3）别名 / 简称 / 模糊称呼映射

虽然已经补了第一版名称映射，但还缺：

- 更完整简称词典
- 常用别名映射
- 模糊消歧

##### （4）非 A 股实体与 A 股影子标的关系识别

例如文本里提到：

- 海外公司
- 未上市公司
- 产品名
- 技术路线

系统现在能过滤掉很多噪音，但还不擅长把这类对象进一步转成“对应受益的 A 股映射链”。

##### （5）多段长文拆分后的上下文连续性

当前已能做一定拆分，但还不够强：

- 一段一只股票时还行
- 一篇里多主题切换时，仍容易漏主标的或误拆

#### 3. 当前输入识别成熟度最准确的分层判断

如果继续按 5 级成熟度看：

1. 不可用
2. 可演示
3. 可系统测试
4. 可稳定内测
5. 可生产使用

当前更接近：

> **3.5 到 4 之间：已经接近“可稳定内测”，但还没到“生产可放心依赖”。**

一句话说就是：

> **已经有了比较像样的输入识别能力，但还不能说已经完全足够。**

#### 4. A 股分时级别量价数据的目标，不应只理解为“拿分钟线”

这一块如果要做得好，建议目标直接定成：

> **建设一个“分层行情数据能力”，而不是只接一个分钟 K 接口。**

原因是后续真正有用的，不只是：

- 1 分钟 / 5 分钟 K 线

还包括：

- 分时成交量
- 分时成交额
- 分时均价
- 开盘后量价结构
- 日内高低点形成过程
- 板块共振与联动

#### 5. 比较好的落地方案：建议分三层推进

##### 第一层：先解决“稳定可拿到分钟级 OHLCV”

建议优先目标：

- 1 分钟 / 5 分钟 OHLCV
- 最近 60 个交易日可回看
- 能按股票批量拉取

这层的供应商建议优先考虑：

1. **AkShare + 东方财富/腾讯/新浪分时接口做调研验证**
   - 优点：便宜，起步快，适合原型验证
   - 缺点：稳定性、限频、长期可维护性一般

2. **Tushare Pro / 聚宽 / 掘金 / Ricequant 这类量化数据服务**
   - 优点：接口规范，A 股生态更成熟
   - 缺点：分钟级权限与历史深度通常要更高套餐

3. **更专业商业源：Wind / iFinD / Choice / 聚源**
   - 优点：稳定、完整、适合长期生产
   - 缺点：成本明显更高

当前如果从“性价比 + 快速落地”角度，建议：

> **先用 AkShare/公开分时源做验证，再尽快切到稳定商业源。**

##### 第二层：把分钟数据沉淀成自己的本地行情仓

不建议每次分析时临时去第三方拉分钟数据。

更好的结构是：

1. 定时采集分钟数据
2. 落本地数据库或 Parquet 文件
3. 自己做统一读取接口

建议本地存储字段至少包括：

- `stock_code`
- `bar_time`
- `open`
- `high`
- `low`
- `close`
- `volume`
- `amount`
- `vwap`（可后算）

这样后续策略层和 LLM 上下文都可以直接读本地标准化数据，不被第三方接口耦死。

##### 第三层：基于分钟数据生成真正有用的日内结构特征

最终直接给模型或策略的，不应只是生的分钟 K 线，而应该是特征摘要，例如：

- 开盘 30 分钟量比
- 早盘冲高回落强度
- 分时均价上方/下方运行时长
- 午后放量突破情况
- 尾盘资金回流强度
- 当日高点形成时间
- 板块内相对强弱排名

也就是说，最优方案不是“分钟数据直喂模型”，而是：

$$
分钟数据 \rightarrow 日内结构特征 \rightarrow 策略判断 / LLM 输入
$$

#### 6. 推荐的现实执行顺序

建议按下面顺序落地：

##### 第一步：选供应商并完成分钟数据可用性验证

重点验证：

- 1 分钟 / 5 分钟数据是否可稳定获得
- 历史回看深度够不够
- 批量股票拉取速度是否可接受
- 成本是否可接受

##### 第二步：新增独立 `intraday_provider`

不要直接把分钟数据塞进现有日线 provider。

更好的做法是新增一层：

- `IntradayMarketDataProvider`

由它统一对外提供：

- `get_intraday_bars()`
- `get_intraday_summary()`
- `get_opening_structure()`

##### 第三步：先做 3 类最有价值日内特征

优先级建议：

1. 开盘半小时量价结构
2. 日内放量突破 / 回落识别
3. 尾盘资金回流 / 走弱识别

##### 第四步：接入股票日评与单票详情页

接入后，系统输出就不再只是日线级：

- 今天涨了多少
- 回撤多少

而会增加：

- 今天是“高开低走”还是“低开高走”
- 放量发生在什么时候
- 分时承接是否健康

#### 7. 一句话结论

##### 关于输入识别：

> 现在已经不能说弱了，已经接近“可稳定内测”；但离真正放心依赖，还差深度研报主标的抽取、简称映射、上下文继承和影子标的关系识别。

##### 关于 A 股分时量价数据：

> 最好的方案不是只找一个分钟 K 数据接口，而是建立“分钟数据源 + 本地行情仓 + 日内结构特征层”的三层方案；短期可先低成本验证，长期应尽快切到稳定商业数据源。

## 二十一、2026-03-08：如果当前目标只是“稳定拿到 A 股分时数据”，可选手段汇总

### 用户当前重点

当前先不继续讨论后续处理逻辑，而是先聚焦一件事：

> **先稳定拿到分时数据。**

这意味着当前判断标准要收敛成：

1. 能不能稳定取到
2. 能不能覆盖 A 股
3. 能不能拿到分钟级 OHLCV / 成交额 / 成交量
4. 成本是否可接受
5. 是否适合后续长期接入系统

### 一、现实可用的 4 类手段

#### 方案 A：公开数据接口聚合（AkShare / 东财 / 腾讯 / 新浪）

这是最低成本起步方案。

常见做法是通过：

- `AkShare`
- 东方财富网页接口
- 腾讯财经接口
- 新浪财经接口

去拿：

- 分时成交明细
- 1 分钟 / 5 分钟 K 线
- 成交量 / 成交额

优点：

- 起步快
- 基本不需要额外付费
- 适合原型验证和小规模测试

缺点：

- 稳定性一般
- 接口可能变更
- 限频和反爬风险存在
- 历史分钟数据深度通常有限
- 不适合长期生产完全依赖

结论：

> **适合“先拿到数据验证系统链路”，不适合当最终长期主数据源。**

#### 方案 B：量化平台 / 数据平台 API

这类是比公开接口更正规、更适合工程化接入的方案。

可考虑：

- `Tushare Pro`
- `聚宽 JoinQuant`
- `Ricequant`
- `掘金 / MyQuant`

一般可以拿到：

- 分钟级 K 线
- 日线 / 复权数据
- 基础行情元数据

优点：

- API 规范
- 工程接入相对稳定
- 文档通常较完整
- 更适合回测和持续采集

缺点：

- 分钟级权限往往需要更高套餐
- 历史深度、并发、频率取决于套餐
- 实时性通常不是最高等级

结论：

> **如果预算有限但希望比公开接口更稳，这是很现实的一档。**

#### 方案 C：券商 / 终端 / 专业金融终端数据

这类属于更专业、更稳定的数据方案。

可考虑：

- `Wind`
- `同花顺 iFinD`
- `东方财富 Choice`
- `聚源`

可提供的通常不只是分钟行情，还包括：

- 分时成交额
- 逐笔 / Level-2（视权限）
- 板块行情
- 资金流
- 公告 / 财务 / 行业数据

优点：

- 稳定性高
- 数据完整度高
- 长期适合生产
- 可直接支持后续策略扩展

缺点：

- 成本高
- 接入流程更重
- 合规和授权要求更明确

结论：

> **如果目标是长期生产级系统，这一类才更像最终方案。**

#### 方案 D：券商量化 / 行情柜台 / 专业实时行情通道

如果后续真的要走接近盘中决策、分钟级刷新甚至更细粒度，就会进入这一层。

例如：

- 券商量化终端接口
- 实时行情订阅服务
- 专门的 Level-1 / Level-2 行情通道

优点：

- 实时性最强
- 更适合盘中策略

缺点：

- 成本与复杂度更高
- 接入、权限、部署要求更高

结论：

> **这不是当前第一步，但如果后面要盘中化，这是迟早要考虑的终局方案。**

### 二、如果当前只看“稳定拿到分时数据”，推荐优先级

如果按你现在这个阶段的真实目标排序：

#### 第一优先级：先验证可用性

建议先上：

- `AkShare + 东财/腾讯分时接口`

用途：

- 验证分钟数据字段是否够用
- 验证系统采集链路
- 验证本地落库和后续读取流程

跑通。

如果预算允许，直接跳到：

#### 第三优先级：后续再考虑更强实时通道

只有当后续目标变成：

### 三、当前最实际的字段要求

### 五、一句话结论

如果现在只问：

那最直接的答案是：

1. **公开接口聚合**：快，但不适合长期完全依赖
2. **量化平台 API**：性价比高，适合中期稳定接入
3. **专业金融终端数据**：最稳，适合长期生产
4. **实时行情通道 / 券商量化接口**：适合后续盘中化阶段

如果现在要实际落地，推荐路径是：

### 二、当前新增接口

1. 分钟 K 线：`/api/intraday/{stock_code}`
2. 逐笔成交：`/api/intraday/{stock_code}/ticks`

分钟接口当前支持：

- `adjust`
- `start_datetime`
- `end_datetime`

返回核心字段包括：

- 时间
- 开高低收
- 成交量
- 成交额
- 振幅
- 涨跌幅
- 涨跌额
- 换手率

逐笔接口当前返回：

- 时间
- 成交价
- 手数
- 买卖盘性质

### 三、为什么这版实现更适合“先稳定拿数据”

这次不是直接裸调公开源，而是加了三层保护：

1. **请求限频**：避免短时间高频连续打接口
2. **失败重试**：单次失败会自动再试
3. **本地缓存回退**：如果远端短时异常，可回退到最近一次成功抓到的数据

所以这版更适合当前阶段：

- 先把分时链路跑通
- 先验证分钟级分析是否够用
- 先验证系统工程稳定性

### 四、关于“东方财富是否有防御”

公开分时源如果底层依赖东方财富这类站点，通常会面临几类现实问题：

1. 请求频率过高时被限流
2. 部分接口返回空数据或间歇失败
3. 站点参数、字段名、页面结构调整后导致上层封装失效
4. 个别时间段出现访问行为校验更严格的情况

这类问题本质上说明：

> **公开源可以用来验证与轻量运行，但不适合作为长期唯一生产依赖。**

### 五、当前阶段建议的解决方式

当前建议采用合规、工程化的稳态方案，而不是去做脆弱对抗：

1. 控制采样频率
2. 做本地缓存
3. 做失败回退
4. 做多源切换预案
5. 后续尽快准备正规分钟数据源

也就是说，现阶段的解决方向是：

> **降低请求强度 + 本地持久化 + 容灾切换**
