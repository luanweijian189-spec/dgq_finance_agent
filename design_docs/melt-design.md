# 融合系统设计文档（dgq_finance_agent × Finance-AI）

版本：v1.0  
状态：设计评审稿（先设计，不落业务代码）  
目标：在最短时间内形成“可用产品 + 可进化策略内核”的A股智能投研与交易辅助系统

---

## 1. 背景与目标

当前已有两套能力：

- `dgq_finance_agent`：偏产品化、可部署、可交互、可运维，已具备消息接入、股票池、每日评估、OpenClaw 指令入口。
- `Finance-AI`：偏策略研究、Agent 协作、方法论扩展、评测与迭代能力更强。

融合目标不是“重写一套系统”，而是构建**双速架构**：

1. **快速上线层（Fast Lane）**：保证今天就能用，支持单股分析、人工干预、稳定日报和告警。
2. **策略进化层（Evolution Lane）**：持续迭代交易逻辑，通过回测/模拟盘/实盘监控不断提升。

核心业务目标（按优先级）：

1. 快速投入使用，保证“输入消息 → 产出可执行分析结论”闭环。
2. 单股可人工分析、可解释、可复盘。
3. 交易逻辑具备鲁棒性（抗噪声、抗情绪、抗单一信号误导）。
4. 面向A股建立可持续迭代机制（周迭代 + 月复盘）。
5. OpenClaw 驱动自动化流程，优先复用现有成熟链路。
6. 信息源以“官方/权威/可追溯”为原则，减少伪信号。

---

## 2. 设计原则

1. **先可用，再最优**：先交付稳定可运行能力，再逐步增强策略复杂度。  
2. **产品层与策略层解耦**：接口稳定，算法可替换。  
3. **证据优先**：任何建议都必须有数据证据链（行情、公告、新闻、逻辑匹配）。  
4. **风险先于收益**：宁可错过，不做失控暴露。  
5. **全链路可观测**：输入、推理、评分、动作、结果可回放。  
6. **人机协同**：系统给建议和评分，人保留关键决策权。

---

## 3. 范围与非目标

### 3.1 本期范围（MVP+）

- 支持微信/文本/手工录入荐股信息。
- 形成结构化股票池与荐股人画像。
- 每日输出单股评分（SQS）、荐股人评分（RRS）、风险提示。
- 支持 `/status /top /who /loop *` 等命令式交互。
- 接入官方可信新闻站点白名单并做语义验证。
- 建立策略评估闭环：离线评测 + 在线观察 + 迭代任务。

### 3.2 非目标（本期不做）

- 全自动实盘下单（先不直连券商交易）。
- 高杠杆、期权、复杂衍生品策略。
- 覆盖全市场全频策略（先聚焦A股 + 日级/小时级决策）。

---

## 4. 融合后总体架构（双速系统）

## 4.1 分层视图

1. **接入层**
	- 消息接入：微信 webhook、手工导入、研究文本导入。
	- 命令接入：OpenClaw 指令（查询、触发评估、触发迭代）。

2. **数据层**
	- 结构化库：股票、推荐、每日评估、荐股人、任务记录。
	- 文本证据库：研究笔记、单股知识文件、新闻快照。
	- 市场数据：行情、量价、基础面指标（先用已接入源）。

3. **分析层（快速上线层）**
	- 规则评估引擎：SQS / RRS / 风险告警。
	- 新闻逻辑验证：白名单站点 + 标题语义匹配 + 内容信号评分。
	- 日报引擎：按天输出股票池追踪报告。

4. **策略层（进化层）**
	- 多Agent策略框架（承接 Finance-AI 方法论）：
	  - 宏观、行业、基本面、择时、资金行为、反身性、风险约束、辩论裁决。
	- 组合决策器：将多Agent观点映射为统一信号（买入/观察/回避）。
	- 评测器：历史回放、滚动窗口评估、漂移检测。

5. **执行与治理层**
	- 任务编排：定时评估、迭代任务、健康检查。
	- OpenClaw 自动化：`init/check/summary` 循环。
	- 可观测性：日志、失败告警、关键指标看板。

### 4.2 核心设计决定

- 保留 `dgq_finance_agent` 作为统一产品入口（API/Web/Agent）。
- 将 `Finance-AI` 能力以“策略插件/服务模块”形式接入，不破坏现有稳定链路。
- 数据模型保持兼容：先扩展字段和中间表，不做高风险重构。
- 所有策略输出统一映射为：`signal + confidence + evidence + risk_flags`。

---

## 5. 统一领域模型（关键实体）

1. `Recommendation`（推荐事件）
	- 谁在何时推荐了什么（股票/方向），原文与提取逻辑是什么。

2. `StockCase`（单股案例）
	- 一只股票在某段时间内的完整证据集合：行情、新闻、逻辑验证、评分轨迹。

3. `SignalSnapshot`（策略信号快照）
	- 每日/每小时的策略结论：方向、置信度、触发因子、风险标签。

4. `RecommenderProfile`（荐股人画像）
	- 胜率、收益分布、回撤特征、风格标签、时间衰减评分。

5. `EvaluationRun`（评估任务）
	- 一次计算执行的输入版本、参数、结果摘要、失败原因。

## 6. 关键业务流程

### 6.1 单股人工分析流程（高优先）

1. 输入：`/status 股票代码` 或 Web 查询。
2. 系统聚合：
	- 最新行情与技术状态；
	- 历史推荐逻辑与验证结果；
	- 官方新闻证据（白名单来源）；
	- 最近风险事件（回撤、波动、流动性恶化）。
3. 输出：
	- 当前等级（A/B/C 或 分值）；
	- 建议动作（观察/小仓位试错/继续持有/减仓规避）；
	- 3 条以内关键证据与反证；
	- 明确失效条件（什么情况下该结论失效）。

### 6.2 每日自动评估流程

1. 收盘后触发调度任务。
2. 对追踪股票计算 SQS，更新 `daily_performance`。
3. 聚合更新 RRS（荐股人可靠性）。
4. 运行新闻逻辑验证并写入证据状态。
5. 生成日报并推送摘要。

### 6.3 策略迭代流程（OpenClaw + Copilot）

1. `/loop init 目标`：定义本轮目标与验收标准。
2. 代码/策略调整后 `/loop check`：自动测试与失败收敛。
3. `/loop summary`：生成下一轮优化提示与风险提醒。
4. 周期复盘：沉淀“有效改动清单 + 无效尝试清单”。

### 6.4 新闻定时扫描与新股发现流程（新增）

目标：不仅更新存量股票结论，还要从新闻中主动发现“候选新股票”。

1. 定时扫描任务（交易时段内）
	- 频率建议：每 10~30 分钟一次；收盘后执行一次全量补扫。
	- 扫描源：官方/权威白名单站点（公告、交易所、主流财经媒体）。

2. 事件抽取与标准化
	- 从新闻标题与正文抽取事件类型：业绩、订单、中标、政策、监管、风险事件等。
	- 提取股票代码/名称并映射标准证券标识。

3. 候选新股打分（Discovery Score）
	- 事件强度：事件是否具备持续性与可验证性。
	- 来源可信度：来源等级与多源一致性。
	- 市场确认度：量价、成交、波动是否支持。
	- 风险扣分：监管、减持、财务异常、流动性不足。

4. 推荐输出与入池策略
	- 分数高于阈值且风险可控：生成“新候选”并入观察池。
	- 分数中等：仅推送提醒，不自动入池。
	- 分数低或高风险：记录为线索，不推荐。

5. 存量股票实时更新
	- 新事件命中已跟踪股票时，增量更新 `SignalSnapshot` 与当日解释文本。
	- 若触发失效条件（逻辑被证伪/风险超阈值），立即降级结论并告警。

6. 人机协同确认
	- 默认新增股票先进入“待确认”队列，由人工一键确认后转入正式跟踪。
	- 对高置信且低风险事件可配置“自动入池（灰度开关）”。

## 7. 交易逻辑框架（鲁棒性设计）

为避免“单一信号误判”，使用四层判定：

1. **宏观与政策层**：是否顺势（政策方向/流动性环境）。
2. **行业与主题层**：是否处于资金与叙事共振区间。
3. **个股基本面层**：业绩、估值、质量、预期差。
4. **交易结构层**：量价结构、筹码状态、风险收益比。

统一输出规则：

- 仅当“方向一致性 + 风险可控 + 证据充分”同时满足时，才给出正向建议。
- 若高收益信号与高风险信号并存，默认降级为“观察”而非“直接行动”。
- 所有建议必须附带止损/失效条件。

---

## 8. 官方可信信息源策略

信息源分级：

1. **一级（官方/权威优先）**：交易所公告、上市公司公告、监管发布、权威财经媒体。
2. **二级（主流媒体）**：覆盖广但需交叉验证。
3. **三级（社区/自媒体）**：仅作线索，不直接作为验证证据。

本系统新闻验证策略：

- 白名单站点抓取；
- 标题语义匹配推荐逻辑；
- 文章正文信号评分；
- 多证据命中后才提升 `logic_validated` 置信度。

扩展策略（新增）：

- 对未在股票池中的标的执行“新闻发现评分（Discovery Score）”；
- 形成“候选新股队列（candidate queue）”，并按阈值触发提醒/入池。

---

## 9. 风险控制体系（必须先行）

### 9.1 信号级风控

- 置信度低于阈值禁止输出“强推荐”。
- 证据链不完整时只输出“信息不足”。

### 9.2 组合级风控（先规则化）

- 单票风险预算上限。
- 行业集中度上限。
- 连续亏损触发冷静期（暂停新增建议）。

### 9.3 系统级风控

- 数据源异常自动降级（行情/新闻断流时给出明确告警）。
- 任务失败重试与人工接管机制。
- 全链路审计日志，支持问题回放。

---

## 10. 评测与进化机制（A股实盘迭代）

采用三阶段评估：

1. **历史回放阶段（T0）**
	- 用历史窗口验证策略稳定性。
	- 指标：胜率、盈亏比、最大回撤、策略漂移率。

2. **模拟运行阶段（T1）**
	- 不下单，仅生成日建议并记录“假设执行结果”。
	- 指标：建议有效率、延迟影响、信号一致性。

3. **小规模真实跟踪阶段（T2）**
	- 人工审核后执行小仓位样本。
	- 指标：真实滑点后收益、风险暴露、回撤控制达标率。

迭代节奏建议：

- 日：监控异常与策略漂移。
- 周：参数微调与失败案例复盘。
- 月：版本级策略评审（保留/降级/淘汰）。

---

## 11. OpenClaw 融合方案

定位：OpenClaw 作为“任务编排与交互入口”，不是策略本身。

职责边界：

- 接收命令、触发流程、返回摘要。
- 执行 `/loop` 迭代闭环。
- 推送日报、告警、健康状态。

不承担职责：

- 不直接定义交易逻辑；
- 不绕过风控下发强制交易动作。

---

## 12. 分阶段落地计划（先快后强）

### Phase 1：7 天快速上线（以 `dgq_finance_agent` 为主）

目标：稳定可用、每天有结果、单股可查可解释。

- 固化输入通道：微信转发 + 手工导入。
- 固化输出通道：日报 + `/status` + 告警。
- 固化质量底线：数据源健康检查 + 基础风控。
- 新增轻量新闻扫描：收盘后全量补扫 + 候选新股提醒。

验收标准：

- 连续 5 个交易日自动评估成功率 ≥ 95%。
- 任一股票查询返回“结论 + 证据 + 风险”。
- 每日候选新股发现结果可回放（来源、事件、分数、处理动作）。

### Phase 2：2~4 周策略增强（引入 Finance-AI 能力）

目标：从规则评估升级为“多Agent共识 + 风险约束”。

- 接入多Agent分析管线。
- 建立统一信号快照模型。
- 上线回放评测报表。
- 新闻扫描升级为盘中准实时（10~30 分钟级），支持自动降级与实时告警。

验收标准：

- 历史窗口指标优于 Phase 1 基线。
- 错误信号（高置信但快速失效）比例下降。

### Phase 3：持续进化（长期）

目标：形成“可持续迭代的交易研究系统”。

- 自动化策略版本管理。
- 月度模型治理与策略淘汰。
- 拓展更多官方数据接口与高质量特征。

---

## 13. 组织与职责分工

1. 产品负责人：定义目标、验收标准、优先级。
2. 策略负责人：维护策略框架与评测方法。
3. 工程负责人：保障系统稳定性与可观测性。
4. 风控负责人：定义风险红线与降级策略。

共识机制：

- 任何“高收益但不可解释”的策略不进入生产层。
- 任何“无法回测复现”的改动不进入默认策略。

---

## 14. 核心指标（北极星 + 护栏）

北极星指标：

- 风险调整后收益提升（相对基线）。

护栏指标：

- 最大回撤不超过阈值。
- 信号失效率不高于阈值。
- 自动任务成功率与数据完整率达标。

过程指标：

- 日报准时率、告警准确率、人工介入次数、迭代收敛速度。

---

## 15. 关键风险与应对

1. 数据漂移与新闻噪声
	- 应对：来源分级、证据交叉验证、低置信降级。

2. 过拟合历史
	- 应对：滚动窗口验证、跨周期评测、版本对照。

3. 自动化误触发
	- 应对：关键动作人工确认、阈值双重校验。

4. 系统复杂度上升
	- 应对：保持“产品层稳定，策略层可插拔”的边界。

---

## 16. 结论

融合方案采用“双速系统”最符合当前阶段：

- 用 `dgq_finance_agent` 保障“立刻可用、稳定输出”；
- 用 `Finance-AI` 驱动“策略持续进化、逻辑持续增强”。

先把**可交付结果**跑稳定，再把**策略能力**做强，是本项目在A股场景下最快、最稳、最可持续的路径。
