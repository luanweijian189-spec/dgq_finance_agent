# dgq_finance_agent 汇报补充问答

日期：2026-03-09

## 1. 关于输入部分的识别

### 1.1 现状

当前输入识别采用“LLM 优先 + 规则回退 + 多格式兼容”的组合方案。

#### 已支持输入来源

- 前端批量导入：`/manual/import`
- API 批量导入：`/api/messages/import_text`
- 单条消息接入：`/api/messages/ingest`
- 微信机器人转发：`/api/connectors/wechat/webhook`
- 手工推荐录入：`/manual/add`
- 研究资料录入：`/manual/research`

#### 已支持识别能力

1. 识别消息类型
   - `recommendation`
   - `tracking_update`
   - `research`
   - `macro`
   - `noise`

2. 识别股票对象
   - 支持 6 位股票代码提取
   - 支持只有股票名称没有代码的识别
   - 支持多股票同时识别

3. 识别荐股信息
   - 荐股人
   - 推荐时间
   - 推荐逻辑摘要
   - 原始消息

4. 支持多种批量导入格式
   - 自由文本
   - 微信常见导出样式
   - JSON
   - CSV

#### 当前识别链路

- 第一层：`LLMInputParserAgent` 做语义理解
- 第二层：`MessageParser` 做规则解析
- 第三层：代码/名称兜底抽取
- 第四层：如果不构成推荐，则转入 RAG/研究资料存储

### 1.2 待优化项

#### 输入识别侧的主要待优化点

1. 股票名称映射仍可增强
   - 当前已支持名称识别，但别名、简称、模糊称呼还有优化空间

2. 置信度与人工审核联动不足
   - 现在有 `confidence`，但尚未形成“低置信度待审核队列”

3. 多股票、多逻辑拆分还可更精细
   - 一条消息中多个标的共用一段逻辑时，目前仍偏统一摘要

4. 前端缺少“识别结果预览”
   - 目前点击导入后直接入库，后续可加“先预览、再确认”模式

5. 输入质量反馈不够可视化
   - 还没有对“哪些消息被判为 research/noise/ignored”做前端明细展示

### 1.3 前端点击“信息输入”后的逻辑和数据链路

#### 前端入口

前端表单在 Dashboard 页，提交到 `/manual/import`。

#### 后端处理顺序

1. 前端表单提交 `raw_text` 和 `default_recommender_name`
2. 路由 `manual_import_form()` 调用 `service.ingest_bulk_text()`
3. `ingest_bulk_text()` 先调用 `_extract_bulk_records()`
4. `_extract_bulk_records()` 按顺序尝试：
   - JSON
   - CSV
   - 自由文本
5. 得到每条记录后，逐条调用 `_parse_recommendations()`
6. `_parse_recommendations()` 内部执行：
   - LLM 解析 `_parse_recommendations_by_llm()`
   - 若失败则走规则解析 `MessageParser.parse_message()`
   - 若仍失败则尝试代码/名称兜底
7. 若能识别为推荐：
   - 调用 `ingest_message()`
   - 建立/复用 `Stock`
   - 建立/复用 `Recommender`
   - 新建 `Recommendation`
   - 写入股票知识库条目
8. 若不能识别为推荐：
   - 调用 `_save_research_note()`
   - 沉淀到研究资料/RAG
9. 返回统计结果到页面

#### 结果类型

- `created`：新增推荐
- `duplicates`：重复推荐
- `ignored`：忽略/未形成有效推荐
- `rag_notes`：转入研究资料的条数

## 2. 实盘数据获取

### 2.1 现状

当前系统采用“免费优先”的组合方案：

- 日线主源：`baostock`
- 盘中主源：`pytdx`
- 盘中回退：`AKShare`
- 组合名：`freebest = pytdx -> AKShare`

#### 已具备能力

1. 日线数据可用于：
   - 日度评估
   - 收益率计算
   - 回撤计算
   - 预测复盘

2. 盘中数据可用于：
   - 1 分钟线
   - 逐笔成交
   - 成交量/成交额
   - 页面绘图
   - 天级维护快照
   - Agent 盘中分析

3. 数据已支持落库
   - `intraday_bars`
   - `intraday_ticks`
   - `stock_daily_maintenance`

### 2.2 缺陷

1. 免费源稳定性有限
   - 接口可能波动、限频、失效

2. 字段口径不完全统一
   - 成交量可能是“股”或“手”
   - 已增加自动识别，但仍存在维护成本

3. 页面刷新与数据刷新耦合较深
   - 进入个股页会触发盘中刷新，响应时间受外部源影响

4. SLA 无法与商业行情源相比
   - 当前更适合投研辅助与汇报，而不是高频交易级场景

## 3. 关于“🔄 一键抓取并更新”的内部数据流和逻辑链路

### 3.1 前端入口

Dashboard 顶部按钮提交到 `/manual/refresh`。

### 3.2 后端触发方式

- 路由 `manual_refresh_form()` 不直接阻塞执行任务
- 它会启动后台线程 `_run_manual_refresh_in_background`
- 页面先返回“已启动后台刷新任务”

### 3.3 核心执行顺序

后台实际执行 `_run_manual_refresh()`，内部顺序是：

#### 第一步：清理无效市场数据

调用 `cleanup_invalid_market_data()`：
- 清理无效周末日评记录
- 清理无效预测记录

#### 第二步：新闻扫描

调用 `run_news_discovery_scan()`：
- 从白名单站点抓资讯
- 识别候选股票
- 计算发现分数
- 保存候选
- 必要时更新已有跟踪对象

#### 第三步：全量日评估

调用 `evaluate_all_recommendations()`：

对所有活跃推荐逐只执行：
1. 获取日线快照
2. 校验推荐逻辑是否被新闻验证
3. 调用 `evaluate_recommendation()` 计算日评分
4. 生成当日 AI 分析摘要
5. 生成/更新预测记录
6. 根据状态机更新推荐状态
7. 所有股票完成后，刷新荐股人评分
8. 生成日报
9. 推送结论更新和日报

### 3.4 最终影响的数据对象

会影响：
- `daily_performance`
- `stock_predictions`
- `recommendations.status`
- `recommenders.reliability_score`
- 新闻候选表
- 首页股票池排序和机会排序

## 4. 目标功能说明

## 4.1 股票评价排序（依赖一键抓取结果）

### 当前实现

系统当前已有两套排序语义：

#### A. 股票池跟踪排序
由 `get_stock_pool_tracking()` 生成，主要展示：
- 最新评分 `latest_score`
- 最新收益 `latest_pnl`
- 荐股人分数 `recommender_score`
- 预测方向/置信度
- 机会分 `opportunity_score`

#### B. 机会排序
由 `list_opportunity_stocks()` 生成，按以下优先级降序：
1. `opportunity_score`
2. `latest_score`
3. `prediction_confidence`
4. `recommender_score`

### 汇报口径

可以表述为：

> 股票排序不是只看涨跌幅，而是综合最新股票质量评分、AI 预测置信度、荐股人可靠度和状态机结果，形成机会优先级排序。

### 待增强项

- 增加盘中维护快照权重
- 增加板块强度/资金流权重
- 增加时间衰减与风格偏好参数

## 4.2 荐股人排序

### 当前实现

荐股人排序由 `get_recommender_list()` 生成，排序规则为：
1. `reliability_score` 降序
2. 推荐次数降序
3. 名称升序

### 评分来源

`refresh_recommender_scores()` 会遍历每个荐股人的历史推荐，提取：
- 最新收益率
- 最大回撤
- 距今天数（时间衰减）

然后调用 `compute_recommender_reliability(outcomes)` 自动计算可靠度。

### 汇报口径

可以表述为：

> 荐股人排序并非主观打分，而是基于历史推荐表现、回撤控制和时间衰减加权的自动评分结果。

## 4.3 手动调整荐股人评分的内部机制

### 当前真实现状

当前系统 **没有完整的前端“手动调整荐股人评分”机制**。

也就是说：
- `reliability_score` 现在主要由系统自动计算
- 每次执行 `refresh_recommender_scores()` 都会覆盖该值
- 因此如果直接改数据库字段，后续会被重算覆盖

### 这意味着什么

如果要支持“人工修正荐股人评分”，不能直接改现有 `reliability_score`，否则系统下一轮刷新会把人工结果冲掉。

### 推荐的正确设计

建议新增以下字段之一：

#### 方案 A：人工加减分
- `manual_score_delta`
- 最终得分 = 自动得分 + 人工校正分

#### 方案 B：人工覆盖分
- `manual_score_override`
- 若存在人工覆盖值，则页面优先展示覆盖值

#### 方案 C：双轨展示
- `system_reliability_score`
- `manual_adjusted_score`
- `final_reliability_score`

### 推荐内部机制

建议采用方案 A：

最终分数公式：

$$
final\_score = clamp(system\_score + manual\_delta, 0, 100)
$$

原因：
- 保留系统自动评分能力
- 保留人工经验修正能力
- 不会丢失原始评分依据
- 适合审计和回溯

### 还应增加的字段

- 调整人
- 调整时间
- 调整原因
- 是否过期

### 汇报口径

可以表述为：

> 当前荐股人分数以系统自动评分为主。后续若要支持人工干预，建议采用“自动评分 + 人工校正分”的双轨机制，既保留算法一致性，也保留业务经验判断。

## 5. 汇报可直接使用的总结

### 输入识别

当前输入识别已经具备 LLM + 规则 + 多格式导入能力，能覆盖荐股消息、跟踪更新、研究资料和噪音消息四类典型场景；下一步重点是低置信度审核、前端预览和名称映射增强。

### 实盘数据

当前实盘数据已具备日线、分钟线和逐笔级能力，能支撑个股盘中看板与 Agent 解读；主要缺陷在于免费数据源稳定性和字段口径一致性仍需持续优化。

### 一键抓取并更新

该按钮本质上触发了一次后台批处理，顺序包含：数据清洗、新闻扫描、全量评估、预测更新、荐股人评分刷新、日报与结论更新。

### 目标功能

股票排序已具备综合排序基础，荐股人排序已具备自动评分基础；手动调整荐股人评分目前尚未形成正式机制，推荐后续采用“系统评分 + 人工校正分”的可审计设计。
